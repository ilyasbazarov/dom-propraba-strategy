## A.1 ПРИНЯТЫЕ РЕШЕНИЯ ПО МОДЕЛИ КЛАССИФИКАЦИИ

### Решение R-01: Смена модели токсичности

**Было (наивная модель):**
`warning при 60 днях без движения → liquidate при 90 днях без движения`

**Проблема:** модель классифицирует **событие** (была ли продажа недавно), а не **экономику** (соответствует ли остаток спросу). Кейс-пример: 100 единиц товара, 1 продажа в квартал → статус «здоровый», но 98 единиц = мёртвый капитал на 25 лет.

**Стало (новая модель):**
Декомпозиция капитала на уровне единиц. Токсичность — не свойство SKU в целом, а свойство **части остатка**. Один SKU может одновременно иметь рабочий запас и мёртвый излишек.

**Зафиксировано:** не откатываться к давности последней продажи как основному критерию.

---

### Решение R-02: Честный ADT через in-stock days

**Проблема (Флаг 1):** ADT, рассчитанный через календарные дни, раздувается при тонких окнах наличия. Пример: продано 40 шт за 1 день из 30 → ADT = 40/30 = 1.33 шт/день → Lost_Revenue = фантазия.

**Решение:**
```
True_ADT = Sold_Qty / in_stock_days   (не Sold_Qty / calendar_days)
in_stock_days = count(is_in_stock_day = 1) за период
```

**Порог валидности:**
- in_stock_days_30 < 7 → adt_confidence = 'LOW' (30-дн окно)
- in_stock_days_90 < 15 → adt_confidence = 'LOW' (90-дн окно)

**Winsorization:** ADT винзоризуется по P95 внутри категории L2. Цель: B2B-всплески (is_b2b_anomaly = 1) не задирают оценку спроса.

**Зафиксировано:** Lost_Revenue и Lost_GP в расчётах использовать только при adt_confidence ≠ 'LOW'.

### Решение R-03: Декомпозиция остатка

**Формулы:**
```python
# Целевой запас
Target_Stock_Qty = True_ADT_90 * Target_Coverage_Days

# Декомпозиция
Excess_Qty      = max(0, Current_Stock_Qty - Target_Stock_Qty)
Working_Qty     = min(Current_Stock_Qty, Target_Stock_Qty)

# Капитал
Working_Capital = Working_Qty * Current_Cost
Excess_Capital  = Excess_Qty * Current_Cost

# Если ADT_confidence = LOW или True_ADT_90 < ε:
# Target_Stock_Qty = 0
# → весь остаток = Excess (либо Dead по доп. условию)
```

### Решение R-04: Классификация stock_health

```python
if True_ADT_90 < epsilon OR Effective_Coverage_Days > 365:
    if LKP_Age_Days > 180 OR Current_Stagnation_Days > 90:
        stock_health = 'DEAD'          # оба сигнала → ликвидация
    else:
        stock_health = 'EXCESS'        # один сигнал → агрессивный trim

elif Effective_Coverage_Days > 180:
    stock_health = 'EXCESS'

elif Effective_Coverage_Days > 90:
    stock_health = 'OVERSTOCK'

else:
    stock_health = 'HEALTHY'
```

**Где:**
```
Effective_Coverage_Days = Current_Stock_Qty / True_ADT_90
   (если True_ADT_90 < ε → Coverage = inf)
```

**Требование двух сигналов для DEAD:**
Попадание в полную ликвидацию требует совпадения:
1. Coverage > 365 ИЛИ ADT < ε при остатке > 0
2. LKP_Age_Days > 180 ИЛИ Current_Stagnation_Days > 90

Логика: если закупщик сам давно не дозаказывал — позиция мертва и по поведению закупки, не только по статистике.

### Решение R-05: KILL расщепляется на два действия

**TRIM** (stock_health = OVERSTOCK или EXCESS, товар живой):
- Сливаем только Excess_Qty до Target_Stock
- Позицию сохраняем в матрице и в e-com каталоге
- Уценка мягкая: −20–30%

**KILL** (stock_health = DEAD):
- Ликвидация всего остатка
- Вывод из матрицы, стоп-кран закупа навсегда
- Лестница ликвидации: B2B-лоты → flash-sale → уценка в зале → списание

### Решение R-06: 5 слоёв каталога как аналитические метки в BI

Слои живут в BI как auto-tagging поле `assortment_layer`, не как управленческая структура.
Управленческие решения — 4 списка: KEEP / KILL / GROW / TEST.

**Правила auto-tagging (role_auto):**
```python
if Revenue_90D / 30 >= locomotive_threshold AND Turnover_Q >= turnover_threshold:
    role_auto = 'LOCOMOTIVE'          # Слой 1: Traffic Spearheads

elif Unit_Margin_T0 / Current_Retail_Price >= 0.40 AND Turnover_Q >= median_turnover:
    role_auto = 'MARGIN_ENGINE'       # Слой 2: Margin Engines

elif sku_age_days <= 120:
    role_auto = 'TEST_POOL'           # Слой 4: новинки на проверке

elif stock_health == 'DEAD' OR stock_health == 'EXCESS':
    role_auto = 'KILL_CANDIDATE'

else:
    role_auto = 'COMPLETER'           # Слой 3: Category Completers
```

**Ручной override:** поле `role_override` перекрывает `role_auto` для исключений.
Пересчёт меток — ежемесячно автоматом.

### Решение R-07: QUARANTINE — маркировка, не удаление

**Причина:** если молча дропать строки, итоги витрины перестают биться с 1С и финансами. Карантин виден, его капитал считается отдельной строкой, totals сходятся.

**В карантине два подтипа:**
- `QUARANTINE_JUNK` — мусор, выводить из матрицы
- `QUARANTINE_FIX` — живой товар, чинить мастер-данные (поставщик, категория)

**Детали** — секция A.4.

## A.3 ПАРАМЕТРЫ МОДЕЛИ

Все параметры зафиксированы. Пересматриваются только с явным обоснованием.

| Параметр | Значение | Обоснование |
|---|---|---|
| `Target_Coverage_Days` | **60 дней** (единый дефолт) | Покрывает цикл заказа + ревью + страховой запас для локальной закупки |
| `Target_Coverage_Days` (импорт/Китай) | **90–120 дней** | Длинное плечо поставки |
| `epsilon` (ADT «фактически ноль») | **0.033 шт/день** = 3 шт/квартал | Ниже — спрос экономически незначим |
| `materiality_threshold` (Excess_Capital) | **3.000 сом** | Ниже — не засорять kill-list, не жечь время команды |
| `adt_min_instock_days_30` | **7 дней** | Меньше — ADT статистически ненадёжен |
| `adt_min_instock_days_90` | **15 дней** | То же для 90-дн окна |
| `adt_winsor_percentile` | **P95 внутри L2** | Гасит B2B-всплески |
| `new_sku_grace_days` | **120 дней** | Новинки в TEST_POOL, не классифицируются |
| `dead_signal_lkp_age` | **180 дней** | Старый LKP = закупщик неявно «похоронил» позицию |
| `dead_signal_stagnation` | **90 дней** | Второй независимый сигнал для DEAD |
| `dead_coverage_threshold` | **365 дней** | При 20–30% стоимости капитала = гарантированное уничтожение стоимости |
| `locomotive_target_count` | **~400 SKU** | Целевое состояние: 400 локомотивов → 45–55% выручки |
| `capital_cost_annual` | **20–30%** | Стоимость собственного капитала в КР |

## A.4 ЛОГИКА QUARANTINE

### Критерии QUARANTINE_JUNK (автоматически):
```python
is_quarantine_junk = (
    Barcode.startswith('INT-')                          # внутренний код без нормального баркода
    AND Primary_Supplier == 'UNKNOWN_SUPPLIER'          # нет поставщика
    AND Category_L1 == '00 БЕЗ КАТЕГОРИИ'              # нет категории
)
```

### Критерии QUARANTINE_FIX (требуют ручной работы):
```python
is_quarantine_fix = (
    (Category_L1 == '00 БЕЗ КАТЕГОРИИ' AND Revenue_90D > 0)   # живой товар без категории
    OR (Primary_Supplier == 'UNKNOWN_SUPPLIER' AND ABC_Revenue_90D == 'A')  # A-класс без поставщика
    OR (Barcode.startswith('INT-') AND Revenue_90D > 0)        # внутренний код с продажами
)
```

### Что делать с QUARANTINE_FIX:
Приоритетная ручная работа перед Фазой 1:
1. Категория «БЕЗ КАТЕГОРИИ» + Revenue_90D > 0 → раскатегоризировать по смыслу товара
2. UNKNOWN_SUPPLIER + A-класс → найти поставщика в 1С или по факту закупок
3. INT-баркод + Revenue_90D > 0 → присвоить нормальный баркод или связать с основным SKU

**Ориентир объёма:** обычно 30–80 позиций требуют реальной ручной работы. Хвост без выручки — оставить в QUARANTINE_JUNK.

## A.5 СЕЗОННЫЕ КАТЕГОРИИ (черновик)

Категории, для которых Coverage_Days считается против ADT пикового квартала, не скользящего:

**Высокая сезонность (не оценивать по стандартным порогам вне сезона):**
- Кровля и кровельные материалы
- Фасадные системы и утепление фасада
- Гидроизоляция кровли
- Садово-дачный инструмент и материалы
- Наружная отделка и краска для фасада

**Умеренная сезонность (оценивать с поправкой +30% к Coverage threshold):**
- Напольные покрытия (пик: весна, осень)
- Обои и декор (пик: весна, осень)
- Антикоррозийные покрытия

**Действие Ильяса на Шаге 6:** подтвердить или дополнить этот список по знанию бизнеса.

## A.6 ЛОГИКА ПИТЧА МАНДАТА

**Когда:** после получения Карты замороженного капитала из Шага 3.
**Аудитория:** владелец бизнеса / ключевые стейкхолдеры.

**Структура питча (1 страница):**

1. **Цифра.** «На складе заморожено Y сом капитала сверх рабочего запаса — это Excess_Capital + Dead_Capital по категориям [таблица]. Этот капитал не нужен для обслуживания текущего спроса — рабочий запас уже вычтен.»

2. **Антивозражение.** Стандартное возражение «нам нужен запас» снято: Working_Capital = необходимый запас = уже выведен из цифры. Y — это только излишек.

3. **Программа.** «Прошу мандат на Программу «Каталог 2.0»: план разморозки капитала и реструктуризации ассортимента под рост GMROI.»

4. **Пакет полномочий (минимальный):**
   - Право уценки и списания в рамках квартального markdown-бюджета (согласовать один раз)
   - Право стоп-крана: вето на reorder любого SKU
   - Право ввода тестовых SKU в рамках capped test-бюджета (~150–200к сом/мес)
   - Ценовые решения по локомотивному слою (KVI-прайсинг требует скорости)
   - Мандат на данные: продавцы фиксируют запросы «нет в наличии»

5. **Контрольные точки.** Квартал 1: X сом высвобождено, GMROI → 0.8+. Квартал 2: Y сом, GMROI → 1.0+. Квартал 4: GMROI → 1.2+.

6. **Sunset clause.** Если через 2 квартала GMROI < 0.8 и высвобождено < Z сом — программа пересматривается. Фиксируется до старта.

**Ключевой принцип питча:** продаём не «дайте полномочия», а «вот X сом замороженных денег, вот план их разморозки, вот метрики». Цифра делает питч неуязвимым к возражению «ну и пусть всё идёт как идёт».

## A.7 ОГРАНИЧЕНИЯ И ФАКТЫ БИЗНЕСА

| Ограничение | Детали | Влияние на программу |
|---|---|---|
| Нет возвратов поставщикам | Bootstrap, реализацию не дают | Трек L только через продажи/уценки/списание |
| Каталог рос без стратегии | Не «под прорабов», «плыть по течению» | Нет исторической логики в структуре — чистый анализ данных |
| Валютная волатильность | LKP_Age > 180 дней → маржа может быть устаревшей | Флагуем is_cost_compromised, используем с осторожностью |
| Нет категорийного менеджера | Владелец программы — Ильяс | Все решения через него, bottleneck на переходный период |
| Слабый кадровый пул | Исполнение операционных задач — ограничение | Регламенты максимально формализуем, снижаем требования к квалификации исполнителя |
| Пост-санационная репутация | Любой сигнал «у них опять проблемы» — риск | Ликвидацию упаковываем как «закрытые предложения клуба», не «тотальная распродажа» |
