# S05_CONVENTIONS · Роли, ритуалы, правила агентов

**Стабильность:** SEMI-STABLE — изменения через ADR или явное решение владельца.
**Версия:** v1.0 · 2026-07-06
**Основа:** метод dom-prorab-catalog (`ilyasbazarov/docs/05_CONVENTIONS.md`)
+ ужесточения ТЗ `_STRATEGY_KICKOFF.md v1.2` + уроки сессии S0.

---

## 1. Роли

| Роль | Модель | Что делает | Чего не делает |
|---|---|---|---|
| **Архитектор** | Opus (дорогая) | Анализ, структура, ADR, брифы, ревью, red-team | Не исполняет массовую обработку материала |
| **Исполнитель-аналитик** | Sonnet (средняя) | Обработка материала, драфты доков и плейбуков по ОДНОМУ брифу | Не принимает архитектурных решений, не выходит за бриф |
| **Applier** | дешёвая модель | Механически применяет патчи | Не интерпретирует, не улучшает |
| **Человек (владелец)** | Ilnar | Источник фактов, владелец решений, исполняет команды, коммитит | Не делегирует финальные решения моделям |

## 2. Anti-improvisation (главный закон)

Если требуемый документ, факт, цифра или решение отсутствуют в контексте —
модель ОСТАНАВЛИВАЕТСЯ и выдаёт:

    CONTEXT GAP: <чего не хватает и почему без этого нельзя>

Запрещено выдумывать правдоподобные значения.

**Ужесточение для стратегии.** Любая цифра (рынок, деньги, конверсии, сроки)
существует только в двух видах:
- **FACT** — подтверждена владельцем или его учётными данными, живёт в S01
  с указанием источника и даты.
- **ASSUMPTION** — явно помечена, живёт в реестре S07, ждёт проверки.

Третьего вида нет. Цифра без метки = нарушение конвенций.

## 3. QNA-контур (контур обогащения контекста)

Принцип: качество выводов модели = функция точности контекста; контекст живёт
в голове владельца и в бизнесе.

1. Каждая сессия ОБЯЗАНА завершиться секцией `QUESTIONS_TO_OWNER`
   (допустимо «нет вопросов»).
2. Вопросы копятся в `S10_OWNER_QA.md` со статусами OPEN → ANSWERED.
3. Закрываются пачками в интервью-сессиях `T-INT-N`: модель задаёт вопросы
   из реестра, владелец отвечает, модель конвертирует ответы в FACT-патчи
   для S01 и закрывает вопросы.

## 4. Riskiest Assumption First

Реестр ASSUMPTIONS в S07 сортируется по риску, не по дате. Допущение, смерть
которого убивает ставку целиком, проверяется первым и самым дешёвым способом.

## 5. Классификация решений (one-way / two-way doors)

Каждое решение в S06 помечается:
- **IRREVERSIBLE** — необратимое или дорогое в откате: полный ADR,
  явный апрув владельца, пауза на обдумывание.
- **REVERSIBLE** — обратимое: решается быстро на уровне сессии, фиксируется
  одной строкой, откатывается дёшево.

Не тормозим 80% решений, не проскакиваем 20% критичных.

## 6. Типы сессий и каденции

| Тип | Что делает | Каденция |
|---|---|---|
| `T-Sx` | Рабочие задачи по брифам (триаж, драфты, плейбуки, модели) | По роадмапу S04 |
| `T-INT-N` | Интервью владельца: OPEN-вопросы из S10 → FACT-патчи в S01 | При ≥5 OPEN или когда OPEN блокирует задачу |
| `T-REV-N` | Ежемесячное ревью: метрики vs tripwires по каждой активной ставке → continue / pivot / kill → патчи S02/S04 | Ежемесячно, календарно, без пропусков |
| `T-RED-N` | Red-team: свежий чат с адверсарным брифом («разнеси стратегию, найди слепые зоны, докажи что обанкротимся») | Обязательно ПЕРЕД ADR-апрувом любого STABLE-дока |

Стратегия, которую не ревьюят по календарю, умирает через квартал —
T-REV пропускать нельзя.

## 7. Session-блок (без него сессия не закрыта)

Каждая рабочая сессия завершается блоком:

    === SESSION BLOCK ===
    SESSION_LOG:       что сделано, входы/выходы, отклонения от брифа
    STATE_PATCH:       изменения для S07 (фокус, ASSUMPTIONS, контрольные цифры)
    NEW_DECISIONS:     решения для S06 (с меткой IRREVERSIBLE/REVERSIBLE)
    NEW_CONVENTIONS:   новые правила для S05 (если появились)
    QUESTIONS_TO_OWNER: вопросы для S10 (или «нет вопросов»)
    === END SESSION BLOCK ===

Один чат = одна порция работы. Переполнение контекста = session-блок
и свежая сессия.

## 8. Доступ к документам

- Агенты читают доки ТОЛЬКО по SHA-pinned raw-URL через curl.
- Владелец даёт SHA коммита в начале сессии; агент верифицирует полученное.
- Ветка main — единственный источник истины.

## 9. Доставка контента в репо (урок S0)

- Архитектор/исполнитель отдаёт контент файлов **готовыми файлами на скачивание**,
  не heredoc-блоками в чате. Владелец кладёт файлы в репо как есть.
- Если файл всё же передаётся текстом: тройные бэктики внутри md-контента
  запрещены (ломают heredoc и рендер); кодовые блоки оформляются отступом
  в 4 пробела; маркер heredoc — уникальный (не `EOF`).
- **Гейт после каждого коммита:** владелец присылает SHA; агент обязан
  скачать файл по SHA и верифицировать целостность (полнота, отсутствие
  обёрток команд в контенте) ДО перехода к следующему шагу.

## 10. Лог решений

S06 — append-only. Решения не переписываются: новое решение помечает старое
как superseded ссылкой на ID.

## 11. Язык и стиль

- Рабочий язык — русский. Термины — простым русским, каждый термин вне
  бытовой лексики обязан существовать в S09_GLOSSARY.
- Структура коммуникации: BLUF (Суть → Причина → Действие),
  короткие предложения, ключевые сущности жирным.
- Плейбуки — по стандарту S08 (язык исполнителя без стратегического мышления).
